# Security & Data Operations

This document covers operational security policies for the Ethical OSINT Tracker.

## Data retention (Issue #15)

Investigation records store the query strings submitted by investigators. Some of
these (email addresses, phone numbers) are PII. To limit indefinite retention:

- The `RETENTION_DAYS` environment variable (default `90`) controls how long
  investigation records are kept.
- A background job (`app/utils/scheduler.py::_purge_retention`) runs once every
  24 hours **and** once at startup, deleting investigations older than
  `RETENTION_DAYS` via `purge_old_investigations()`.
- Set `RETENTION_DAYS=0` (or any non-positive value) to disable automatic
  purging.

### Why query strings are not hashed

`app/utils/crypto.py` contains `hash_if_sensitive()`, which is intentionally
**not** wired into investigation creation. The query string is the human-readable
label an investigator relies on to know which target a result belongs to; hashing
it (one-way SHA-256) would make the tool's own case views and dashboard unusable.
Instead, PII exposure is mitigated by:

1. **Per-user scoping** — the dashboard and "recent investigations" views
   (`list_recent`, `count_all`, `count_by_kind`) are scoped to the current user,
   so users never see other users' targets.
2. **Retention purging** — old records are deleted per the policy above.

## Cross-user isolation (Issue #7)

All case detail/edit/delete/export routes enforce `case.owner_user_id ==
current_user.id` unconditionally and `abort(403)` otherwise. Async export jobs
are also tagged with `user_id` and verified on status/download.

## API key storage (Issue #5)

API keys are encrypted at rest with Fernet symmetric encryption. Set
`API_KEYS_FERNET_KEY` to a key generated by:

```bash
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

If the key is not set the application logs a warning and falls back to storing
keys unencrypted — **do not run production without this key set**.

## Committed photos — git history remediation (Issue #9)

Three real photos were previously committed under `uploaded_files/`. They have
been removed from the working tree and `uploaded_files/` is now in `.gitignore`,
so they will not reappear in new commits. EXIF review found **no GPS
coordinates**, but device make/model (OPPO A18, Toshiba webcam) and capture
timestamps were exposed.

**Deleting the files from the tree does not remove them from git history.**
Because rewriting history changes every commit hash and requires a force-push,
this step must be performed by a repository administrator on the default branch,
after obtaining contributor consent:

```bash
# From a fresh clone (git-filter-repo removes the origin remote as a safety):
pip install git-filter-repo
git filter-repo --path uploaded_files/ --invert-paths
git remote add origin <repo-url>
git push --force --all
git push --force --tags
```

All collaborators must then re-clone, as history will have diverged. This was
deliberately **not** automated in the fix branch because a history rewrite +
force-push on the shared default branch is a destructive, consent-gated admin
action.
